# Rename Folder Skill — 文件夹代码批量重命名整理

当用户请求对某个文件夹下的代码进行重命名整理时（如"整理这个文件夹"、"重命名这些demo"、"统一命名"、"rename <路径>"），执行以下步骤。

## 设计原则

- **读代码内容，不猜文件名。** 文件名可能是临时起的，代码才是真相。
- **难度层级统一框架。** 命名格式：`{难度}-{功能描述}.py`，不保留原始序号。
- **父文件夹即上下文。** 父文件夹名（如"正则"、"代理"）已说明主题域，文件名不再重复主题词。
- **用户确认后再执行。** 先展示计划，用户确认后才重命名。

## 难度层级定义

| 层级 | 适用场景 | 典型特征 |
|------|----------|----------|
| `基础` | 该主题的第一个接触点、核心 API 入门 | 单一函数调用、最简示例、无复杂逻辑 |
| `进阶` | 在基础之上的参数变化、模式组合、边界处理 | 多参数组合、标志位切换、异常处理、非默认行为 |
| `实战` | 面向真实场景的完整应用 | 有目标 URL、写入文件、解析实际网页、多步骤串联 |

判断规则：
- 同一文件夹下，按学习递进关系分配层级。先学的是基础，后学的是进阶或实战。
- 如果文件夹只有一个文件且内容简单 → `基础`
- 如果代码中有真实 URL 或文件 I/O → 优先考虑 `实战`
- 同级可以有多个文件，用功能差异区分（如 `基础-匹配与搜索.py`、`基础-分组提取.py`）
- 不够 3 个文件时，可以只用 1-2 个层级，不必强行凑三级

---

## 步骤 0：自动判定模式

列出目标文件夹下所有 `.py` 文件，根据文件名特征自动进入 A / B 两路之一：

### 模式 A：从零命名（占位符命名）

**判定条件**：文件夹中 ≥ 1 个 `.py` 文件名匹配以下任一模式（不区分大小写）：

- `demo*.py`
- `^\d+\.py$`（纯数字，如 `1.py`、`2.py`）
- `test*.py`、`练习*.py`、`new*.py`、`temp*.py`

**处理范围**：只处理匹配上述模式的文件。已有语义化命名的文件（如 `aiohttp_socks5.py`）默认跳过，除非用户明确要求也处理。

### 模式 B：校准统一（已有命名但不准确/不统一）

**判定条件**：文件夹中大部分 `.py` 已有语义化命名，但存在以下问题之一：

- 命名风格不统一（中英混杂、有的有前缀有的没有）
- 命名不够精确（描述过大或过小，如 `spider.py` 实际只做了验证码识别）
- 命名冗长或包含无关信息

用户通常会说"统一命名"、"重新命名"、"整理一下这些文件名"。

**处理范围**：所有 `.py` 文件均为候选。对每个文件判断是否需要重命名——如果现有名称已经准确、简洁且风格统一则保留。

---

## 步骤 1：确认输入

向用户确认目标路径（若已提供则跳过）。模式已在步骤 0 自动判定，告知用户当前走的是哪种模式。

## 步骤 2：读取并分析代码

对每个候选文件：

1. 读取完整代码内容
2. 提取关键信息：
   - import 了什么（判断技术栈）
   - 核心函数/方法调用（判断功能）
   - 是否有真实 URL、文件路径（判断是练习还是实战）
   - 代码复杂度（行数、控制流、函数定义数）
   - **（仅模式 B）** 旧文件名中含有什么线索（如 `aiohttp_http_auth.py` → 提示了 aiohttp + HTTP 认证）
3. 记录每个文件的功能摘要（一句话）

## 步骤 3：分配难度层级

1. 按代码从简到繁排序（行数、import 数量、控制流复杂度）
2. 最简单的 1-2 个文件 → `基础`
3. 有明显参数变化、模式切换的 → `进阶`
4. 有外部 URL、文件 I/O、多步骤组装 → `实战`
5. 如果所有文件复杂度相当，按功能语义区分而非强行分三级——可以全用 `进阶`

## 步骤 4：生成功能描述

功能描述规则：
- 中文，≤10 字
- 动词/名词开头，描述这段代码**做了什么**，不是"第几课"
- 同级多个文件时，每个文件的功能描述必须互斥（能一眼区分）
- 不重复父文件夹名中已有的主题词
- **（仅模式 B）** 旧名中有价值的关键词（如技术名 `aiohttp`、`Selenium`）可酌情保留

示例：
```
模式 A（父文件夹: 正则）
demo1.py → 基础-match与匹配对象.py
demo2.py → 基础-分组捕获.py

模式 B（父文件夹: 代理）
aiohttp_http&_http_auth.py → 进阶-aiohttp代理认证.py
requests_socks5.py          → 进阶-SOCKS5代理.py
```

## 步骤 5：展示计划并确认

向用户展示重命名计划表格：

```
📁 <文件夹路径>（模式 A/B）

| 原文件        | → 新文件              | 一句话功能    |
|--------------|----------------------|--------------|
| demo1.py     | → 基础-xxx.py        | ...          |
| old_name.py  | → 进阶-yyy.py        | ...          |
```

**模式 B 特别标注**：对建议保留原名的文件也列入表格，标注"（保留）"。

然后询问用户：**"确认执行重命名？"**

用户可以：
- ✅ 确认 → 执行步骤 6
- ✏️ 修改某个命名 → 回到步骤 4 调整
- ❌ 取消 → 不执行任何操作

## 步骤 6：执行重命名

逐文件重命名：

```bash
mv "<原路径>" "<目标路径>"
```

> 注：优先用 `git mv`，若文件未被 git 追踪则退化为普通 `mv`。

执行完成后，向用户报告结果：

```
已完成：重命名 8 个文件（保留 2 个）
基础 2 | 进阶 4 | 实战 2
```

## 注意事项

- 仅处理 `.py` 文件。其他类型文件跳过。
- 如果新文件名已存在，在新文件名后加 `_2` 后缀避免覆盖，并告知用户。
- 不修改代码内容，只重命名文件。
- 如果用户提供的路径不存在，报错并提示检查路径。
- 如果目标文件夹下没有需要处理的文件，告知并结束。
- 路径分隔符统一用 `/`。
- 目录名以 `-` 开头时（如 `-course`），mv 命令中需加 `./` 前缀避免被解析为 flag。
