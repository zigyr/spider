# Snippet Extract Skill — 项目经验片段提取

当用户请求"经验化"某个文件时（如"经验化这个文件"、"提取为经验"、"snippet"、"/经验化 <路径>"），执行以下步骤。

## 设计原则

- **用户控制判断，skill 执行机械操作。** 哪个文件值得提取、功能描述怎么写——由用户决定。文件路径读取、注释写入、重命名、移动——由 skill 执行。
- **文件名即索引。** 重命名后的文件名应让用户扫一眼就知道这段代码做什么，不需要打开看注释。

## 步骤 1：确认输入

向用户确认两个信息（若用户已同时提供则跳过）：

1. **文件路径**（必填）：要经验化的 `.py` 文件的仓库相对路径
2. **功能描述**（必填）：一句话说明这段代码做了什么，用于生成文件名和注释

若用户未提供功能描述，询问用户。**不要自行推断功能描述**——用户最清楚这段代码的核心价值是什么。

## 步骤 2：读取源文件

读取目标文件的完整内容，了解其代码结构和依赖（import 语句、关键函数等）。

同时对源文件所在的项目目录做基本了解（同一目录下还有哪些文件），以便理解该经验的上下文。

## 步骤 3：确定来源路径

源文件的当前相对路径即为"原项目"来源。例如：

- 文件在 `-projects/视频/cn.xgroovy.com/check_range.py`
- 来源路径记录为 `-projects/视频/cn.xgroovy.com/`

如果文件所在目录本身就是项目根（如 `cn.xgroovy.com/`），来源路径取到项目目录层级即可，不需要具体到文件名。

## 步骤 4：生成新文件名

根据用户提供的功能描述，生成**中文"我做了什么"模式**的文件名。

规则：
- 中文描述，一句话说清功能
- 动词开头或动名词结构（如"检测..."、"从...提取..."、"下载..."）
- 保留 `.py` 扩展名
- 避免英文（除非是专有名词如 URL、HTTP、m3u8）
- 长度控制在 20 字以内

示例：
- 功能："检查服务器是否支持 HTTP Range 请求" → `检测服务器是否支持Range请求.py`
- 功能："从剪切板读取数据并提取 URL" → `从剪切板提取URL.py`

## 步骤 5：写入/更新 docstring

在源文件的**最顶部**（shebang 之后、import 之前）写入或更新 docstring，格式如下：

```python
"""
原项目 -projects/视频/cn.xgroovy.com/
功能   检查服务器是否支持 HTTP Range 请求
"""
```

规则：
- `原项目`：使用正斜杠 `/` 的相对路径，指向来源项目目录（以 `/` 结尾）
- `功能`：用户提供的功能描述（完整句子，不用逗号分隔）
- 如果文件已有该格式的 docstring，更新而非重复
- 如果文件已有其他格式的顶部注释，替换为此格式
- docstring 与后续代码之间空一行

## 步骤 6：移动文件

将文件从源路径移动到 `_recipes/` 下：

```
_recipes/<新文件名>
```

- 使用 `git mv` 移动（如果仓库是 git 仓库），以便 git 追踪重命名
- 如果 `_recipes/` 下已存在同名文件，在文件名后加数字后缀（`_2`、`_3`）避免覆盖，并告知用户
- 如果源文件所在目录在移动后变为空目录，告知用户可手动清理

## 步骤 7：验证与报告

移动完成后，验证目标文件存在且内容完整，然后向用户报告：

```
已经验化：
  源文件: -projects/视频/cn.xgroovy.com/check_range.py
  目标:   _recipes/检测服务器是否支持Range请求.py
  功能:   检查服务器是否支持 HTTP Range 请求
```

简洁即可，不用多余解释。

## 注意事项

- 仅处理 `.py` 文件。其他类型文件提示用户不支持。
- 不修改代码逻辑，只添加/更新 docstring 和移动文件。
- 如果用户提供的文件路径不存在，报错并提示检查路径。
- 如果文件已在 `_recipes/` 下，提示用户该文件已经是经验片段，询问是否需要更新注释。
