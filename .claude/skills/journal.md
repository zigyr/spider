# Journal Skill — 爬虫学习日志自动记录

当被调用时（由 CLAUDE.md 自动触发，或用户手动调用），执行以下步骤。

## 步骤 1：获取变更文件列表与代码差异

按优先级尝试：

```bash
# 优先：已暂存的文件列表 + 实际代码差异
git diff --cached --name-only
git diff --cached --stat       # 看每个文件的改动量分布
git diff --cached -p           # 完整 diff 内容（核心！）

# 其次：未暂存的工作区改动
git diff --name-only
git diff --stat
git diff -p

# 兜底：最近的 commit
git log --oneline -n 3
git show --stat HEAD
git show -p HEAD
```

只关注以下文件类型：`.py`、`.md`、`.txt`、`.json`、`.html`。忽略 `__pycache__/`、图片文件、`.gitignore`、`journal/` 自身的变更。

如果变更文件列表为空，跳过本次记录，不写入任何内容。

**关键原则：分析基于代码内容，不信任文件名。** 文件名可能是用户随手起的（如 `demo5.py`、`test.py`），必须通过阅读 diff 内容和实际代码来确定脚本的真正用途。

## 步骤 2：模块归类

根据文件路径前缀自动归类：

| 路径前缀 | 模块名 |
|----------|--------|
| `01-basics/` | `01-basics` |
| `02-course/` | `02-course` |
| `03-dll/` | `03-dll` |
| `04-projects/` | `04-projects` |
| `_notes/` | `笔记` |
| 根目录或无法归类 | `其他` |

如果一次变更涉及多个模块，每个模块生成一条独立条目。

## 步骤 3：阅读代码变更并理解实际内容

**核心原则：以代码逻辑为准，文件名仅作参考。** 文件名不可信——用户可能随手起名（`demo5.py`、`test1.py`、`练习.py`），必须从代码中判断真实用途。

### 分析流程

1. **先看 diff 内容**（从步骤 1 的 `git diff -p` 获取）：了解每处新增/删除/修改了什么逻辑
2. **再读关键文件的完整代码**（特别是 diff 大的文件）：理解上下文和全局结构
3. **提取真实信息**，无视文件名

### 阅读策略

- 对于**新增**的 `.py` 文件：先看 diff（即全部新增内容），重点关注 import 语句、函数定义、类名、核心循环逻辑
- 对于**修改**的 `.py` 文件：从 diff 中定位改动行，再打开文件读取上下文（改动函数的前后 20-30 行）
- 对于新增/修改的 `.md` 文件：快速浏览主题即可
- 如果变更文件过多（>15 个），优先读取 diff 量最大的 3-5 个 `.py` 文件，其余概括

### 从代码中提取（不从文件名提取）

- **主题**：不看文件名，从 import、函数调用、请求 URL 模式、解析逻辑中推断（如导入了 `m3u8`、`Crypto.Cipher.AES` → M3U8 解密下载；导入了 `asyncio`、`aiohttp` → 异步并发请求）
- **技术点**：实际用到的库/方法/模式（如 `aiohttp`, `ThreadPoolExecutor`, `Crypto.Cipher.AES`, `xpath`, `正则`）
- **变更意图**（修改类文件）：改了什么逻辑、为什么改（如"修复了 cookie 过期判断"、"适配网站新 DOM 结构"、"从同步 requests 重构为异步 aiohttp"）

### 日志质量检查

写完一条日志前，问自己：**如果别人看了这条日志，能知道这段代码在做什么、而不需要看代码本身吗？** 如果不能，说明分析不够深入，需要重新阅读代码。

## 步骤 4：去重检查

在写入前，检查 `journal/YYYY-MM-DD.md` 是否已存在，若存在则检查是否已有相同主题+相同文件的条目。若已有，跳过不重复写入。

## 步骤 5：生成并写入日志

### 文件路径

```
journal/YYYY-MM-DD.md
```

### 条目格式

若当天文件不存在，创建新文件并写入：

```markdown
# YYYY-MM-DD

## 模块名 | 简短主题（一句概括）
- **文件**: `相对路径`（新增/修改）
- **内容**: 一句话描述：做了什么事、学到了什么
- **技术点**: `技术1`, `技术2`, `技术3`
```

若当天文件已存在，在已有内容末尾追加一条新的 `## 模块名 | 主题` 条目。

### 格式要求

- **标题**: `## 模块名 | 主题` — 主题控制在 15 字以内，**必须反映代码实际功能，不能照搬文件名**（文件叫 `demo5.py` 但内容做的是模拟登录 → 标题用"模拟登录实现"而非"demo5"）
- **文件**: 使用仓库相对路径，标注变更类型（新增/修改/删除），多个文件用 `、` 分隔
- **内容**: 一句话，描述"做了什么、学到了什么"，从学习视角写，基于代码分析而非文件命名推测
- **技术点**: 1-3 个关键技术或库名，用反引号包裹，逗号分隔
- 条目之间空一行

### 示例

```markdown
# 2026-07-17

## 01-basics | 文件路径处理
- **文件**: `01-basics/文件与目录/输出放在脚本同级目录的文件夹下.py`（新增）
- **内容**: 练习将下载输出放到脚本同级目录，使用 os.path 动态获取脚本所在路径
- **技术点**: `os.path.dirname`, `os.path.join`, `os.makedirs`

## 04-projects | AGE动漫 AES-128 解密修复
- **文件**: `04-projects/age-anime-v1/decrypt.py`（修改）
- **内容**: 修复 AES-CBC 解密 key 提取逻辑，适配目标网站 DOM 结构调整
- **技术点**: `Crypto.Cipher.AES`, `re.compile`, M3U8 流解析
```

## 步骤 6：完成确认

写入完成后，向用户简要告知（一句话即可）："已记录今日学习日志：{主题1}、{主题2}"。
